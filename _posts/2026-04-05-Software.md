## 0. Background

### 0.1 Project Overview

The **Limb Motion Recognition and Assistant** system is a distributed rehabilitation platform jointly developed by UTAD (Portugal) and Jilin University (China) under the DSD 2025–2026 programme.

The system captures joint motion data from wearable IMU sensors worn by rehabilitation patients, processes that data through AI-based motion recognition on a central server, and delivers real-time feedback to patients and historical analysis to medical professionals.

|------|-------|----------------------|
| **S1** | Sensor | IMU hardware, sensor firmware, wireless protocol (IF1 specification) |
| **S2** | Sensor | Data acquisition pipeline, mock data generator, IF1 contract owner |

| **M1** | Monitor | Mobile app (patient interface), real-time visualisation |
| **M2** | Monitor | Clinical web dashboard (doctor/admin interface), 3D reconstruction |

### 0.4 Deployment Contexts

The system must support two deployment contexts:

| Context | Characteristics |
|---------|----------------|
| **Clinical Setting** | Medical staff assist with sensor setup. Stable network. Real-time data upload. Supervised sessions. |
| **Home Setting** | Patients operate independently. Network may be unstable. System must guide setup and tolerate intermittent connectivity with local buffering. |

## 3. Interface Definitions

### 3.1 IF1 — Sensor to Server (S2 → V1)

| Field | Description |
|-------|-------------|
| Session ID | Unique identifier for result tracing |
| Motion Data | Structured sensor values (JSON schema) for preprocessing and inference |
| Status Information | Flags for invalid or incomplete input detection |
| Sampling Rate | Data sampling frequency agreed between S1/S2 and V1 |
| Timing / Order | Sequence order information for packet ordering |
| Error Codes | Standardised error codes for communication failures |
| Error-Handling Rules | Rules for invalid payloads and communication problems |

**Owners:** S2 and V1 (co-owned)  
**Freeze Deadline:** End of Sprint 1

### 3.2 V1-V2 Internal Contract

| Output Block | Description |
|--------------|-------------|
| Recognition Result | Recognised motion result in structured form |
| Comparison Result | Angle comparison output (when available) |
| Processing Status | Success, partial, failed, or other agreed status values |
| Warnings / Errors | Low-confidence markers, failure information |
| Metadata | Confidence score, model version, timestamp |

**Owners:** V1 and V2 (co-owned)

### 3.3 IF2 — Server to Monitor (V2 → M1/M2)

| Field | Description |
|-------|-------------|
| Patient Data | Session records, progress trends |
| Exercise Plan | Prescribed exercises, target angles |
| Alerts | Posture alerts, anomaly notifications |
| Reports | Generated analysis reports |
| Authentication Tokens | JWT or equivalent for session management |

**Owners:** V2 and M1 (co-owned)  
**Freeze Deadline:** End of Sprint 2

| NFR-02 | End-to-end feedback latency (sensor → feedback) | < 500ms |
| NFR-03 | Session data storage | Support > 1000 sessions per patient |
| NFR-04 | Concurrent users | Support 100+ simultaneous sessions |

### 6.2 Reliability Requirements

| ID | Requirement | Target |
|----|-------------|--------|
| NFR-05 | System uptime | 99.5% availability |
| NFR-06 | Data loss tolerance | Zero data loss for completed sessions |
| NFR-07 | Offline mode | Graceful degradation with local buffering |

### 6.3 Security Requirements

| ID | Requirement |
|----|-------------|
| NFR-08 | All data transmission shall use TLS encryption |
| NFR-09 | Passwords shall be stored using bcrypt or Argon2 hashing |
| NFR-10 | JWT tokens shall expire after 24 hours |
| NFR-11 | Patient health data shall comply with GDPR/regional regulations |

### 6.4 Usability Requirements

| ID | Requirement |
|----|-------------|
| NFR-12 | Mobile app shall provide step-by-step guidance for home users |
| NFR-13 | Clinical dashboard shall display all key metrics on first screen |
| NFR-14 | Error messages shall be clear and actionable |

### 6.5 Maintainability Requirements

| ID | Requirement |
|----|-------------|
| NFR-15 | All interfaces shall be documented in English |
| NFR-16 | Code shall include inline comments for cross-team readability |
| NFR-17 | API contracts shall be versioned |

## 8. Assumptions and Constraints

### 8.1 Assumptions

- IF1 will be frozen by the end of Sprint 1
- IF2 will be frozen by the end of Sprint 2
- Expected recovery values will be provided by the medical team or derived from literature
- Patients will have access to smartphones (iOS/Android) with Bluetooth LE
- Clinical settings will have stable WiFi/network infrastructure

### 8.2 Constraints

- IMU sensors have limited battery life (~8 hours continuous use)
- Bluetooth range limited to ~10 metres
- Home network reliability varies by location
- GDPR compliance required for patient health data
